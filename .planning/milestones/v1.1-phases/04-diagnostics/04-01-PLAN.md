Create `~/.claude/graphiti/diagnose.py` — a standalone diagnostic script using the same Python venv as graphiti-helper.py. The script tests each pipeline stage independently and prints a structured report to stdout.

**Stage probes (in order, each independent):**

1. **probe_docker**: Check Docker containers are running.
   - Run `docker ps --filter name=graphiti --format '{{.Names}} {{.Status}}'`
   - PASS: both `graphiti-neo4j` and `graphiti-mcp` show "Up"
   - FAIL: container missing or unhealthy

2. **probe_neo4j**: Check Neo4j is reachable from the host.
   - HTTP GET `http://localhost:7475` (Neo4j browser port mapped in docker-compose.yml)
   - PASS: HTTP 200
   - FAIL: connection refused or non-200

3. **probe_graphiti_health**: Check Graphiti MCP health endpoint.
   - HTTP GET `http://localhost:8100/health`
   - PASS: HTTP 200
   - FAIL: connection refused or non-200

4. **probe_mcp_session**: Test MCP session initialization.
   - POST `http://localhost:8100/mcp` with JSON-RPC initialize request (same payload as MCPClient._initialize)
   - PASS: response contains `mcp-session-id` header or valid JSON-RPC result
   - FAIL: connection error, timeout, or error response
   - Log: full response headers and body (this is the first non-trivial step)

5. **probe_mcp_write**: Test MCP add_memory tool call directly.
   - Initialize MCP session (reuse MCPClient from graphiti-helper.py by importing it, OR reimplement the 3 HTTP calls)
   - Call `add_memory` tool with: `{"name": "diagnostic-probe", "episode_body": "Diagnostic canary write at {ISO_TIMESTAMP}", "group_id": "global", "source": "text", "source_description": "diagnostic-probe"}`
   - PASS: response contains `result` key with no error
   - FAIL: response contains `error` key — log the full error message
   - Log: full JSON-RPC response

6. **probe_mcp_read**: Test MCP search_memory_facts tool call.
   - Call `search_memory_facts` with: `{"query": "diagnostic canary", "group_ids": ["global"], "max_facts": 5}`
   - PASS: response contains results (may include the canary from probe 5)
   - FAIL: error response
   - Log: full response

7. **probe_project_scope_write**: Test writing to a project scope specifically.
   - Detect current project using `graphiti-helper.py detect-project` (run via subprocess)
   - Call `add_memory` with `group_id` set to `"project:{detected_project_name}"`
   - Episode body: `"Diagnostic project-scope canary at {ISO_TIMESTAMP}"`
   - PASS: no error in response
   - FAIL: log the error — this is likely where DIAG-02 breaks
   - Log: detected project name AND full response

8. **probe_project_scope_read**: Verify the project-scoped write is retrievable.
   - Call `search_memory_facts` with `group_ids` set to `["project:{detected_project_name}"]`
   - Query: `"diagnostic project-scope canary"`
   - PASS: results contain the canary text
   - FAIL: empty results despite probe_project_scope_write succeeding — indicates a write-but-not-indexed problem

9. **probe_helper_add_episode**: Test graphiti-helper.py add-episode command directly (WITH stderr visible).
   - Run via subprocess: `~/.claude/graphiti/.venv/bin/python3 ~/.claude/graphiti/graphiti-helper.py add-episode --text "Helper probe canary at {ISO_TIMESTAMP}" --scope global --source diagnostic`
   - Capture BOTH stdout AND stderr (do NOT use `2>/dev/null`)
   - PASS: exit code 0 and no stderr output
   - FAIL: non-zero exit code OR stderr contains error messages — log everything
   - This is the critical probe: if this fails but probe_mcp_write succeeds, the bug is in graphiti-helper.py's cmd_add_episode wrapper

10. **probe_hook_simulation**: Simulate a hook's write path with stderr visible.
    - Run the same command that capture-change.sh runs, but WITHOUT `2>/dev/null` and WITHOUT `&`:
      `~/.claude/graphiti/.venv/bin/python3 ~/.claude/graphiti/graphiti-helper.py add-episode --text "File Write: /tmp/diagnostic-test.txt" --scope "project:{detected_project}" --source "change-hook"`
    - Capture stdout and stderr
    - PASS: exit code 0, no stderr
    - FAIL: stderr contains error messages — this is exactly what `2>/dev/null` is hiding
